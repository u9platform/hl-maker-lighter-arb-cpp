# Baseline 2026-03-27

Environment:

- host: local development machine
- bridge interpreter: `.bridge-venv/bin/python`
- sample size: `5`
- commands measured via `scripts/perf_baseline.py`

Results:

- `hl_orderbook`
  - mean: `1068.31ms`
  - p50: `893.40ms`
  - max: `1632.49ms`
- `lighter_orderbook`
  - mean: `2195.67ms`
  - p50: `2202.61ms`
  - max: `2313.72ms`

Interpretation:

- the Python bridge plus current venue clients is functional, but not low-latency
- Lighter orderbook fetch is currently the biggest bottleneck in this stack
- before optimizing strategy logic further, we should reduce or replace the bridge path for market data

Immediate next optimizations:

- replace bridge subprocess spawning with a persistent bridge process
- move HL and Lighter market data collection into native C++
- keep the bridge only for authenticated trading calls until native signers are ready

Follow-up native public-market-data sample:

- command: `./native_baseline 3`
- `native_hl_orderbook`
  - mean: `1700.23ms`
  - max: `4038.55ms`
- `native_lighter_orderbook`
  - mean: `46.09ms`
  - max: `61.09ms`

Updated interpretation:

- Lighter market data improved dramatically once Python and subprocess startup were removed
- HL market data is still slow in the current native implementation and needs connection reuse or websocket migration
- the hot path is now uneven: Lighter BBO is acceptable for a first native pass, HL still is not
